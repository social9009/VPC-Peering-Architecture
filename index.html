Create VPCs

VPC-A Mumbai 10.100.0.0/16 + IGW. VPC-B N.Virginia 10.200.0.0/16 (no IGW). Must have non-overlapping CIDRs.

02
Subnets & RTs

Public subnet 10.100.0.0/24, private 10.100.11.0/24. VPC-B private 10.200.11.0/24. Route tables for internet & local routes.

03
Launch EC2

EC2-A (public, bastion), EC2-B (private), EC2-C (private). Security groups: allow SSH/ICMP from VPC-A CIDR on target EC2-C.

04
Peering + routes

Create inter-region peering request → accept in peer region → add routes in both private route tables pointing to peering connection.

Troubleshooting & Pitfalls

🚫 Ping still failing after peering?
✔️ Check BOTH route tables have routes to remote CIDR.
✔️ EC2-C security group must allow ICMP from 10.100.0.0/16.
✔️ Peering status must be 'Active'.
✔️ Confirm you use private IP of EC2-C.
🔑 SSH from EC2-A → EC2-B permission denied
➜ Use SSH agent forwarding (ssh -A) or copy .pem file to EC2-A.
➜ Verify EC2-B SG allows SSH from VPC-A CIDR.
➜ Same key pair for all instances simplifies lab.
🌐 Peering request not visible in other region
Switch to correct region (N.Virginia) and check "Pending acceptance". Ensure VPC IDs are correct and no CIDR overlap.

Key Concepts Recap

VPC Peering
Private, encrypted link across regions/accounts
Non‑transitive
Peering A↔B + B↔C ≠ A↔C
Route tables mandatory
Each VPC needs explicit route to peer CIDR
No overlapping CIDRs
10.100.0.0/16 and 10.200.0.0/16 are safe
Bastion host
EC2-A provides internet access to private subnets
Clean‑up — Terminate EC2s, delete peering, delete VPCs to avoid costs. VPCs themselves are free but EC2 & data transfer incur charges.
